# LoopSail 仓库开发约定

本文件适用于 LoopSail 插件源码的维护。开始修改本仓库前，先完整阅读 `DEVELOPMENT.md`，再检索根目录 `开发经验.md` 中与当前模块、错误现象和设计领域相关的既有条目。

LoopSail 2.x 的长期协议约束是：Worker 必须是可见的会话内 `loopsail:worker` 子 Agent；Coordinator 只能通过 `prepare-step`/`finalize-step` 编排；所有持久协议使用 v2 和 Draft-07；禁止恢复嵌套 `claude -p`、`--json-schema` 或直接手改 Plugin cache 的路径。修改任一层时必须同步检查 commands、Skill、Agent、hooks、运行时验证器、checked-in schemas、模板与测试。

根目录的 `开发经验.md` 是插件维护者使用的工程经验库。`plugins/loopsail/skills/loopsail/templates/经验记录.md` 则是 `/loopsail:init` 写入目标项目、由 Coordinator 维护的运行期复盘模板。两者用途和维护者不同，不得混用；本文件也不改变 Worker、Coordinator 或目标项目 `经验记录.md` 的既有职责边界。

## 经验记录触发条件

开发过程中真实遇到以下任一情况时，必须更新 `开发经验.md`：

- 失败、绕路或反复尝试背后存在非显而易见的根因；
- slash command、Skill、Schema、Coordinator、Worker 或测试等协议层发生遗漏、不同步或隐含耦合；
- 权限、安全、Git、初始化、配置、兼容性、缓存、安装或发布流程出现具有复用价值的典型问题；
- 发现已有证据确认、但当前任务范围内无法修复的设计缺陷；
- 既有经验条目不准确、不完整，或同一问题出现了新的触发条件、影响或验证方式。

以下内容不应记录：

- 普通拼写错误、语法错误或一次性命令输入失误；
- 根因和修复都显而易见、没有复用价值的常规测试失败；
- 未经证实的猜测、尚无证据的风险设想；
- 与 LoopSail 无特定关系的通用编程知识；
- 凭据、隐私数据、私有绝对路径、大段原始日志或其他敏感信息。

顺利完成且没有命中触发条件时，不要为了留下记录而编造经验。

## 记录流程

1. 开发前先搜索 `开发经验.md` 的标题、分类、影响范围和关联位置，避免重复踩坑。
2. 开发中保留足以解释现象、根因和验证结果的非敏感证据，但不要把操作流水直接写入经验文档。
3. 完成实现和验证后、给出最终答复前，执行一次“经验判定”。命中触发条件时，在同一份改动中更新 `开发经验.md`。
4. 写入前再次搜索相似条目。同一问题应更新原条目，补充新的证据、适用范围或解决结果，不得创建重复条目。
5. 已完成修复的条目标记为“已解决”，并写明实际验证依据。已确认但当前无法修复的问题标记为“待处理”，记录影响和可执行的下一步；后续修复时原位更新为“已解决”。
6. 更新旧条目时保留原始问题背景。不得为了让当前修复显得完整而删除仍有学习价值的失败路径或历史约束。

`开发经验.md` 是知识库，不是项目规范或任务输入。如果某条经验形成新的长期工程约束，应同时更新 `DEVELOPMENT.md`、README、Schema、测试或其他对应的权威来源；经验条目负责记录原因和教训，不能代替正式规范。

如果当前请求仅允许只读诊断、明确禁止修改文档，或上级指令限制了可编辑路径，不要擅自更新文件；应在最终答复中指出建议沉淀的内容。除此之外，插件开发任务命中触发条件时，可将 `开发经验.md` 视为随实现一并维护的文档。
