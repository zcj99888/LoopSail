# LoopSail 插件开发经验

本文档用于沉淀维护 LoopSail 插件源码时真实遇到的困难、绕路、设计缺陷、根因、修复方式和可复用原则，供后续维护者学习和检索。

这里记录的是插件开发经验，不是 LoopSail 运行目标项目时的任务复盘。后者使用 `plugins/loopsail/skills/loopsail/templates/经验记录.md`，由 Coordinator 按运行协议维护。不要在两个文档之间复制或混合条目。

## 维护规则

- 使用中文撰写，必要的技术标识、文件名和命令保持原样。
- 只记录真实发生、具有证据且能够复用的问题；顺利完成的常规修改不留条目。
- 每个条目只描述一个问题，内容保持简洁，并使不了解当时上下文的维护者也能理解。
- 写入前先搜索相似标题、现象、根因和关联位置。同一问题应更新原条目，不重复追加。
- “已解决”必须包含实际验证依据；不能仅凭代码看起来合理就宣称解决。
- “待处理”只收录已经确认的问题，必须写明现有证据、影响和可执行的下一步，不得把推测包装成结论。
- 修复“待处理”问题时原位更新状态、修复方案和验证依据，并保留原始问题背景及有学习价值的绕路。
- 本文档不是设计规范或待办系统。形成长期规则时同步更新对应的权威文档、Schema 或测试；待处理项仍应在项目采用的任务系统中跟踪。
- 不记录凭据、隐私数据、私有绝对路径、大段原始日志或与复盘无关的敏感内容。

## 经验条目

## 2026-07-20 · JSON Schema 方言必须与实际消费者能力对齐

- 状态：已解决
- 分类：协议 | 兼容性
- 影响范围：Worker 结构化输出、全部 checked-in schema、首次任务执行
- 现象与触发条件：首个任务尚未执行实现时，嵌套 Claude CLI 的 `--json-schema` 即报无法解析 Draft 2020-12 meta-schema URI，两次尝试都没有有效 Worker 结果。
- 绕路：只把已安装 cache 中的 schema URI 手工改成 Draft-07，表面上消除了当前副本的报错，但权威源码和其他 schema 仍保持旧方言。
- 根因：协议作者按较新的 JSON Schema 方言生成文件，却没有以 Claude Code 内部 validator 的真实能力作为兼容边界；同时 schema 方言散落在多个文件，没有生成一致性测试。
- 修复方案或下一步：Protocol v2 由 `protocol.py` 统一生成 Draft-07 schemas，使用精确 URI `http://json-schema.org/draft-07/schema#`、`definitions` 和本地引用；运行时使用标准库严格验证器，不再把 schema 交给嵌套 CLI。
- 验证依据：37 项 unittest 已断言所有生成/checked-in schema 完全一致且不含 `$defs`，两项 `claude plugin validate --strict` 均通过；正式安装 2.0.4 后的新空项目通过字面 `/loopsail:init → /loopsail:validate → /loopsail:run-once` 冒烟，request/result/event/attempt/step v2 流程完整，相关 session 与独立 transcript 中没有 `--json-schema` 或 meta-schema 解析错误。
- 可复用原则：Schema 的权威版本不是“最新草案”，而是所有实际消费者共同支持的方言；方言、生成器、运行时校验器和 fixtures 必须由测试锁定为一个协议单元。
- 关联位置：`scripts/protocol.py`、`references/*.schema.json`、`tests/test_loopsail.py`

## 2026-07-20 · Plugin cache 不是可维护的修复目标

- 状态：已解决
- 分类：发布 | 工具链
- 影响范围：Marketplace 更新、已安装插件版本、问题复现与发布可靠性
- 现象与触发条件：故障诊断过程中，活动 cache 的 schema 被直接修改后与源码仓库及 Marketplace clone 不一致；当前会话可能暂时恢复，但重新更新、换 profile 或换项目后修复会消失。
- 根因：把 Claude Code 的版本缓存误当成源码工作树。cache 是安装产物，没有版本化审阅，也会被正常插件更新替换。
- 修复方案或下一步：只修改权威仓库、提升插件版本并提交推送；使用相同 Claude profile 正式执行 Marketplace update 和 project-scope plugin update，绝不手改 clone/cache。真实冒烟发现的问题依次发布为 2.0.1 至 2.0.4 补丁，均走相同正式链路。
- 验证依据：源码与文档已经删除 cache 修补路径并加入发布检查；Marketplace 与 CorpusLint project-scope 安装均由 `.claude-ds` profile 的官方 CLI 更新，活动版本 2.0.4 已启用并通过最终字面 slash 新会话冒烟。
- 可复用原则：调试安装产物只能用于定位，不能作为修复交付；任何持久修复都必须走“源码 → 版本 → Marketplace → 安装 → 新会话”链路。
- 关联位置：`README.md`、`DEVELOPMENT.md`、`plugins/loopsail/.claude-plugin/plugin.json`

## 2026-07-20 · 嵌套非交互 CLI 会把 Worker 过程变成黑箱

- 状态：已解决
- 分类：架构 | 测试
- 影响范围：Worker 可观察性、中断恢复、输入输出绑定、失败诊断
- 现象与触发条件：原 Worker 通过 `claude -p` 配合 `capture_output` 启动，外层会话只能看到最终 stdout/stderr；首轮失败日志中的 Worker 进程元数据为空，用户看不到具体工具调用过程，也没有独立会话 transcript 可查。
- 根因：Coordinator 把模型执行当作子进程，并同时承担启动、超时、结构化输出和日志截取；这绕开了 Claude Code 原生子 Agent 的 UI、transcript 与 Agent hooks 生命周期。
- 修复方案或下一步：改为插件级 `loopsail:worker` 会话内子 Agent，Supervisor 前台串行执行 prepare/Agent/finalize；hooks 绑定 request/agent、执行工具 guard、写脱敏事件并捕获严格 result。
- 验证依据：集成测试覆盖绑定、guard、事件脱敏/截断、一次输出纠正、缺失结果、重复失败和幂等 finalize；2.0.4 字面 `/loopsail:run-once` 的实际工具序列为固定 Bash actions、唯一前台 Agent 和 finalize，独立 transcript 存在，13 条事件通过字段与脱敏检查，Coordinator 独立验证后生成任务提交。
- 可复用原则：需要过程可见性和生命周期安全控制时，应使用宿主的原生 Agent 抽象；结构化结果协议不能替代可观察的执行过程，也不能只依赖 Worker 自报信息。
- 关联位置：`agents/worker.md`、`hooks/hooks.json`、`scripts/hook.py`、`scripts/loopsail.py`

## 2026-07-20 · PreToolUse 安全通过不等于宿主权限授权

- 状态：已解决
- 分类：安全 | 兼容性
- 影响范围：`dontAsk` 模式下 Worker 的所有文件写入和命令执行
- 现象与触发条件：2.0.0 首次真实子 Agent 冒烟中，guard 对合法 `Write` 没有返回拒绝，但宿主仍以权限不足阻止调用；单元测试只验证了“不拒绝”，没有覆盖宿主需要的明确授权结果。
- 根因：PreToolUse hook 的安全判定与 Claude Code 权限判定是两个阶段；空输出只表示 hook 不否决，并不授予 `dontAsk` 下尚未预授权的工具调用。
- 修复方案或下一步：安全通过时明确返回 `permissionDecision: allow` 和事件名，拒绝分支继续返回 `deny`；以 2.0.1 补丁正式发布。
- 验证依据：回归测试断言允许分支的 hook 输出；后续真实 Worker 的 `Write` 与 `Bash` 均成功执行且事件分别记录 allowed/succeeded。
- 可复用原则：权限 hook 的“允许、拒绝、无意见”是三种不同语义；自动化模式必须逐项测试宿主实际消费的输出，而不是只测试本地 guard 返回值。
- 关联位置：`scripts/hook.py`、`tests/test_loopsail.py`

## 2026-07-20 · 一次纠正必须重申完整的结果约束

- 状态：已解决
- 分类：协议 | 测试
- 影响范围：SubagentStop 的单次纠正机会、严格 Worker JSON 捕获
- 现象与触发条件：真实 Worker 首次结果同时包含 Markdown 围栏和绝对 `changed_files` 路径；第一次校验只报告 JSON 文档错误，Worker 去掉围栏后仍保留绝对路径，第二次校验按设计生成协议失败结果。
- 根因：严格验证器按顺序 fail fast，而纠正反馈只重申了当前首个错误，没有提醒 Worker 同时复核路径等后续约束。
- 修复方案或下一步：Worker 契约明确要求项目相对 POSIX 路径；任何首次协议错误的纠正反馈同时重申原始 JSON、绑定字段和相对路径约束，以 2.0.2 补丁发布。
- 验证依据：回归测试锁定纠正反馈；真实 retry 的合法结果使用 `changed_files: ["smoke.txt"]` 并被 stop hook 捕获，Coordinator 随后成功提交。
- 可复用原则：当协议只允许一次纠正且验证器 fail fast 时，错误反馈既要指出当前错误，也要附带最容易连带违反的完整输出清单。
- 关联位置：`agents/worker.md`、`scripts/hook.py`、`scripts/protocol.py`

## 2026-07-20 · 控制文件保护必须覆盖读取而不只是修改

- 状态：已解决
- 分类：安全 | 协议
- 影响范围：Worker 对任务清单、经验文件、Git 内部和其他 Coordinator 控制文件的访问
- 现象与触发条件：2.0.2 真实事件日志显示 Worker 在已获得完整 request 后仍直接 `Read` 了 `TASKS.json`；guard 当时只对 `Edit/Write` 应用 protected_paths。
- 根因：实现把“用户配置的只读保护路径”和“Worker 不应访问的 Coordinator 控制路径”合并成一组，但读取阶段只特殊处理了 `.loopsail`。
- 修复方案或下一步：将内置控制路径与用户 protected_paths 分开；除绑定 request 外，文件工具对任务文件、经验文件、Git 内部及 LoopSail 控制路径一律拒绝，用户 protected_paths 仍保持只禁止修改；以 2.0.3 补丁发布。
- 验证依据：单元测试覆盖直接读取 `TASKS.json` 和经验文件；2.0.3 最终新会话事件只出现绑定 request、声明的 context file、目标文件和脱敏 Bash 调用，没有控制文件读取，任务仍成功提交。
- 可复用原则：控制面的机密性边界与普通写保护是不同策略，必须分别建模并对 Read/Glob/Grep/Edit/Write 各自验证。
- 关联位置：`scripts/guard.py`、`agents/worker.md`、`tests/test_guard.py`

## 2026-07-20 · Slash 命令中的 action 名不能依赖模型自行路由

- 状态：已解决
- 分类：兼容性 | 测试
- 影响范围：全部 `/loopsail:*` 用户入口、固定 Coordinator action 的权限边界
- 现象与触发条件：底层 action 冒烟成功后，首次字面 `/loopsail:init` 冒烟仍把文案中的“run doctor”解释成 `Skill(loopsail:doctor)`；`dontAsk` 正确拒绝后，模型又尝试普通 Bash 探测，而没有调用 frontmatter 中允许的固定 doctor action。
- 根因：命令只列出了 allowed-tools，却用自然语言简称引用 action；工具白名单约束权限，但不会替模型消除 `Skill`、嵌套 slash 与 Bash action 之间的路由歧义。
- 修复方案或下一步：所有 slash 命令明确禁止 Skill、嵌套 slash 和探索性命令，并把每个 action 绑定到 `${CLAUDE_PLUGIN_ROOT}/skills/loopsail/scripts/loopsail.py slash <action>` 的字面 Bash 调用；以 2.0.4 补丁发布。
- 验证依据：静态测试要求每个命令包含明确路由规则；2.0.4 新安装项目依次执行字面 `/loopsail:init`、`/loopsail:validate`、`/loopsail:run-once`，实际 run-once 工具序列为 Bash/Bash/Bash/Agent/Bash，无 Skill 调用，任务提交成功。
- 可复用原则：面向模型的工具编排既要声明允许集合，也要明确唯一调用形式；入口级冒烟不能用直接调用底层实现代替。
- 关联位置：`commands/*.md`、`tests/test_skill_bundle.py`
